    01 — Le point de départ

    Le chat d'aujourd'hui : ce qu'il fait déjà, et ses vraies limites

    Soyons honnêtes sur le point de départ, sinon le reste sonne faux.

    Ce que l'IA-chat fait DÉJÀ (et ce n'est pas rien) :

    • elle ne fait pas que « parler » : elle crée des fichiers, analyse tes pièces jointes (PDF, images, tableurs), génère des tableaux, des graphiques, des images ;
    • elle exécute du code pour calculer, nettoyer un fichier, produire un graphe (les modes type « analyse de données ») ;
    • elle navigue le web quand il faut une info à jour ;
    • et elle a de la mémoire : mémoire persistante (elle retient des choses d'une conversation à l'autre), espaces « projets », instructions personnalisées.

    Donc non, ce n'est ni « juste des mots », ni une amnésie totale. Disons-le clairement.

    Les vraies limites (précises) :

    1. La mémoire existe, mais elle est limitée et non maîtrisée. Elle est opaque (tu ne vois pas vraiment ce qu'elle retient ni pourquoi), résumée (donc lossy : elle garde une version compressée, pas le détail exact), propre à l'outil (pas portable d'un produit à l'autre), et surtout non versionnée (pas d'historique, pas de retour en arrière, pas « qui a changé quoi quand »).
    2. Le travail reste piloté tour par tour, dans une session éphémère. C'est toi qui conduis chaque étape ; quand la session se ferme, l'espace de travail s'évapore ; ce qu'elle produit, tu dois le récupérer et le ranger toi-même.
    3. Elle est coupée de ton environnement réel. Elle ne connaît que ce que tu lui colles. Elle ne touche pas, d'elle-même, à tes fichiers métier ni à tes applications.
    En une phrase

    Le chat d'aujourd'hui fait déjà beaucoup ; mais sa mémoire est limitée et hors de ton contrôle, il travaille tour par tour dans une session éphémère, et il est coupé de ton environnement réel.

    02 — Le saut

    Le saut : modèle → agent, et la connexion au réel

    2.1 Modèle ≠ agent (les bons mots)

    Attention au raccourci. Un agent, ce n'est pas « une autre IA ». C'est une question d'équipement autour du même type de cerveau.

    • Le modèle (le LLM, pour Large Language Model) = le cerveau : il prédit du texte à partir de ce qu'on lui donne. C'est le même genre de modèle qui répond dans le chat et qui anime un agent.
    • L'agent = ce même modèle placé dans un dispositif qui lui donne quatre choses :
      1. une boucle : il planifie → agit → observe le résultat → recommence, jusqu'à atteindre l'objectif ;
      2. des outils : lire et écrire des fichiers, lancer des commandes, appeler des applications ;
      3. un environnement : tes fichiers, ton dossier de travail ;
      4. un objectif + ta validation.
    En une phrase

    Le modèle, c'est le cerveau. L'agent, c'est le même cerveau équipé (boucle + outils + environnement) et autonome. Pas une autre IA — une IA mise au travail.

    2.2 Comment un agent agit, concrètement

    Tu lui donnes un objectif (« prépare le compte-rendu de ce client à partir de ces notes »). Il découpe en étapes, agit (ouvre les fichiers utiles, rédige, range), vérifie son propre résultat, corrige, et te montre pour validation. Tu restes le pilote ; lui tient le volant sur les étapes fastidieuses.

    2.3 Se brancher sur tes outils métier : API et MCP

    C'est ici que ça devient puissant — et c'est un poil plus avancé, mais accessible.

    • API (interface de programmation) = la façade par laquelle un logiciel expose ses services à d'autres logiciels. Chaque service précis est un endpoint (« point d'entrée ») : une porte dédiée, du type « donne-moi la liste des factures » ou « crée une fiche client ». Presque toutes les apps que tu utilises (CRM, compta, agenda, drive…) ont une API.
    • MCP (Model Context Protocol) = un standard ouvert (apparu fin 2024) qui sert d'adaptateur universel entre un agent et tes outils/données. Au lieu d'un branchement bricolé pour chaque app, un protocole commun : tu « branches » l'outil une fois, l'agent sait s'en servir.

    Ce que ça change vraiment : tu peux connecter un agent aux applications que tu utilises déjà dans ta routine métier, pour qu'il lise et agisse sur tes vraies données — pas seulement sur ce que tu colles à la main.

    Et le truc qui surprend : un agent peut souvent se débrouiller seul pour créer le pont — découvrir les endpoints d'une application existante, aller lire sa documentation en ligne, écrire le code de la connexion, voire créer une clé ou un compte qui n'existe pas encore. (Avec les garde-fous évidents : autorisations, authentification, et ta validation — il agit dans le périmètre que tu lui donnes.)

    En une phrase

    Via API et MCP, l'agent sort de la fenêtre de chat et se branche sur tes outils métier réels — et il peut souvent construire ce branchement lui-même.

    03 — La mémoire intègre

    git : d'où ça vient, ce que c'est, bien s'en servir

    3.1 D'où vient git (et pourquoi ça compte)

    git a été créé en 2005 par Linus Torvalds — le créateur de Linux. Le contexte : l'outil qu'utilisait le noyau Linux (BitKeeper) leur a été retiré du jour au lendemain. Torvalds avait besoin, en urgence, d'un système capable de gérer des milliers de contributeurs travaillant en parallèle sur le même code, partout dans le monde. Il l'a conçu autour de quatre exigences qui expliquent tout ce que git sait faire :

    1. Distribué : chacun a une copie complète de tout l'historique sur sa machine. Pas de serveur central obligatoire, pas de point de panne unique.
    2. Rapide, même sur d'énormes historiques.
    3. Intègre : chaque version est scellée par une empreinte cryptographique — impossible d'altérer le passé en douce. L'historique est une preuve.
    4. Non-linéaire : on peut travailler sur des branches parallèles et les fusionner, pour que des centaines de gens avancent sans s'écraser les uns les autres.

    (Le nom « git » ? Torvalds, fidèle à lui-même, a plaisanté en disant qu'il nommait ses projets d'après lui — « git » étant aussi de l'argot britannique pour « sale type ».)

    Le point profond : git a été pensé pour la collaboration massive, distribuée, traçable, avec droit d'expérimenter. C'est exactement ce dont un agent a besoin pour travailler proprement. D'où le couple agent + git.

    3.2 Ce que git fait — les concepts, expliqués

    • Dépôt (repo) = un dossier suivi par git.
    • Commit = un point de sauvegarde daté et commenté (tu écris pourquoi tu as fait ce changement).
    • Versionné / historique = la suite des commits. « Versionné » veut dire : chaque état est conservé, horodaté, attribué, et tu peux revenir à n'importe lequel. Rien ne se perd.
    • Branche = une ligne de travail parallèle. Tu pars de l'état actuel, tu expérimentes à côté sans toucher l'original ; si ça marche, tu fusionnes ; sinon, tu jettes la branche, et l'original n'a jamais bougé.
    • Réversibilité = revenir à un état antérieur en un geste.
    • Traçabilité = savoir qui a changé quoi, quand, et pourquoi.

    Et rappel essentiel : git ≠ GitHub. git tourne sur ta machine. GitHub (ou GitLab, Bitbucket, ton propre serveur…) n'est qu'un cloud optionnel où copier ton dépôt. git en local suffit déjà à tout ce qui précède.

    C'est, concrètement, la fin du rapport_final_v2_vraiment_final(3).docx.

    3.3 Pour aller plus loin : bonnes pratiques git + l'esprit agile

    Ces concepts ne sont pas du folklore de développeur : ce sont les bonnes pistes pour utiliser ces outils efficacement, y compris hors dev.

    Côté git :

    • Commits fréquents et atomiques : une idée = un commit (plus facile à relire, à annuler).
    • Messages clairs : décris le pourquoi, pas seulement le quoi.
    • Branches pour expérimenter : teste sans risque.
    • .gitignore : exclus ce qui ne doit jamais être suivi/partagé (secrets, gros fichiers).

    L'esprit agile (les bases, expliquées) : une manière de travailler qui consiste à avancer par petits incréments livrables plutôt qu'en un grand bloc, avec :

    • des itérations courtes (l'itératif = avancer par petites boucles « fais un bout → vérifie → ajuste → continue », au lieu de tout d'un coup) ;
    • du feedback fréquent ;
    • une « definition of done » (se mettre d'accord sur ce que veut dire « fini ») ;
    • une rétrospective (qu'est-ce qu'on améliore au prochain tour ?).

    Pourquoi c'est connecté à l'agentique : un agent travaille mieux en itérations bornées, avec un « done » clair et une validation fréquente. Tu le pilotes exactement comme tu piloterais une mini-équipe agile : petits pas, vérifie, recommence.

    En une phrase

    git, c'est une mémoire intègre, versionnée et réversible, née pour la collaboration massive ; et bien s'en servir, c'est adopter l'esprit agile (petits pas, « done » clair, feedback) — pile ce qui rend un agent efficace.

    04 — La documentation

    La doc qui fait tourner le tout

    On l'oublie toujours, et c'est pourtant ce qui sépare un agent qui patine d'un agent qui file droit.

    4.1 Il y a DEUX niveaux de documentation

    Quand on dit « documentation », les gens imaginent un seul type de document. En réalité il y en a deux, qui ne s'écrivent pas pareil :

    • Les docs pour les humains : narratives, lisibles, faites pour cadrer le travail, expliquer, transmettre, embarquer quelqu'un. (Ce que tu imagines spontanément.)
    • Les docs pour les « sachants » / l'agent : structurées, concises, factuelles, faites pour être consultées par la machine — règles, conventions, « où trouver quoi ». Ce n'est pas de la prose : c'est de l'information adressable.

    Les deux coexistent et sont complémentaires.

    4.2 Indexation + arborescence : pour que l'agent ne relise pas tout

    Le problème : si tout est entassé dans un gros document, l'agent doit tout parcourir pour trouver une info — c'est lent, ça coûte des tokens, et ça sature sa mémoire de travail (limitée, voir §5).

    La solution :

    • une arborescence (une structure en dossiers/fichiers thématiques) ;
    • un index : un sommaire + des rappels d'orientation du type « pour la facturation, va voir tel fichier ».

    Ainsi l'agent navigue vers le bon document au moment où il en a besoin, au lieu de tout avaler. (Et accessoirement, ça aide aussi les humains.)

    4.3 Le journal de suivi

    Un journal (un log de bord) qui retrace, dans le temps, ce que l'agent a fait, quand, et ce qui a été livré / produit.

    • Pourquoi : tu gardes la vue d'ensemble, tu peux auditer, reprendre où tu en étais, et prouver ce qui a été fait.
    • Complément de git : git dit ce qui a changé dans les fichiers ; le journal raconte les décisions et les livrables. Les deux ensemble = une traçabilité complète, comme un vrai carnet de bord.
    En une phrase

    Une bonne doc, c'est deux niveaux (humains / agent), une arborescence indexée pour naviguer sans tout relire, et un journal de suivi qui garde la trace de ce qui a été fait et livré.

    05 — Une autre catégorie

    Pourquoi agent + git + bonne doc = une autre catégorie

    Pas « le chat en mieux ». Une autre catégorie. Voici pourquoi, sans raccourci.

    D'abord une précision qui corrige une idée fausse répandue : l'agent ne relit pas tout ton dossier à chaque fois. Sa mémoire de travail (sa fenêtre de contexte) est limitée et temporaire. Il ne charge pas tout d'un coup : il va chercher, fichier par fichier, seulement ce dont il a besoin pour la tâche en cours — guidé par l'arborescence indexée du §4. C'est précisément parce que sa mémoire vive est limitée qu'on externalise la mémoire durable dans les fichiers + git.

    Du coup, la combinaison donne :

    • Une mémoire persistante ET maîtrisée. Tes fichiers + git forment une mémoire durable, explicite, versionnée, portable, que TU contrôles (à l'opposé de la mémoire opaque du chat, §1). L'agent la consulte sélectivement.
    • Traçable et réversible. Chaque étape est un commit (§3) : tu peux annuler, comparer deux versions, reprendre.
    • Le droit d'expérimenter. Tu branches pour tester une variante sans rien casser.
    • Connecté au réel. Via API/MCP (§2), l'agent agit sur tes vraies données et apps.
    • Tu capitalises. Ton dossier-métier s'enrichit au fil du temps ; le journal de suivi garde le fil.

    La bascule, c'est : une IA autonome qui travaille dans TON environnement, avec une mémoire que tu maîtrises et que tu peux prouver, branchée sur tes outils, et qui capitalise.

    6.4 Alors : abonnement, API, ou local ?

    Règle générale : l'API est moins chère pour l'usage léger/occasionnel ; l'abonnement devient rentable dès l'usage soutenu (il plafonne la facture) ; le local ne se rentabilise qu'à fort volume et/ou exigence de confidentialité.

    Chiffré (exemple Claude) :

    • Usage léger — ~30 questions/mois. Sur l'API Sonnet : ≈ 0,40 $/mois. → L'API écrase l'abonnement à 20 $ ; payer un forfait n'a aucun sens.
    • Usage intensif quotidien — un agent de code tournant toute la journée peut brûler des dizaines de MTok/mois. À 50 MTok entrée + 5 MTok sortie sur Opus : ≈ 375 $ en API. → L'abonnement Max 20× (200 $) est nettement moins cher et inclut Claude Code.
    En une phrase

    Usage léger → API au token (souvent < 1 $/mois). Usage humain intensif → abonnement (Pro/Max). Confidentialité maximale ou très gros volume → local (matériel à amortir).

    Cette grille de coût se lit en face de la grille de risque du chapitre suivant : le moins cher pour l'usage léger (grand public / API) n'est pas le plus protecteur ; le plus protecteur (local) est le plus cher à l'entrée. C'est l'arbitrage central.

    07 — Tes données

    Le sujet sérieux : tes données, ce qui transite, le risque réel

    C'est LE point qu'on doit traiter franchement.

    7.1 Ce qui part vraiment, à chaque requête

    Un LLM hébergé (Claude, ChatGPT…) tourne sur les serveurs du fournisseur. À chaque requête, partent chez lui :

    • ta question (le prompt) ;
    • le contexte de la conversation ;
    • tout fichier/image que tu joins ;
    • en mode agent : le contenu des fichiers que l'agent ouvre + les résultats des actions qu'il relit.

    7.2 Le piège (et la clé) de l'agentique

    Le chat n'envoie que ce que tu colles. L'agent, lui, décide quoi lire dans le périmètre que tu lui ouvres.

    Tout ce que l'agent doit lire pour faire le job part chez le fournisseur.

    Donc la vraie question n'est pas « est-ce dangereux ? » mais « quel périmètre je lui ouvre ? ». Dossier de templates anodins → rien de sensible ne part. Dossier clients en clair → ça part. Avec un modèle local (§6.3), rien ne sort.

    7.3 Les trois mondes (à ne pas confondre)

    Entraîné sur tes données ?Conservation
    Grand public (apps ChatGPT/Claude, formules perso)Oui par défaut — sauf opt-outVariable, jusqu'à plusieurs années si tu laisses
    API / business & entrepriseNon, jamaisCourte (jours) → voire zéro (ZDR)
    Modèle localNon (ça ne sort pas)Toi seul

    7.4 L'angle qu'on oublie (et qui change tout) : construire l'outil ≠ traiter la donnée

    C'est rarement dit, et c'est pourtant décisif. Il faut distinguer deux usages de l'IA :

    1. Utiliser l'IA pour traiter tes données → tes données transitent (cas du §7.1).
    2. Utiliser l'IA pour construire un outil → ce qui transite, c'est le code de l'outil (sa logique), pas tes données métier.

    Le scénario concret : tu te sers d'un agent pour bâtir un petit outil qui absorbe une friction métier. Pendant la construction, ce qui part chez le fournisseur, c'est le code — pas tes données clients. Ensuite, l'outil tourne en huis clos (sur ta machine / ton serveur) pour traiter les données confidentielles. Et si cet outil n'appelle pas lui-même un LLM distant, tes données confidentielles ne sortent jamais.

    (« Huis clos » = en vase clos, sans que rien ne sorte vers l'extérieur.)

    La conséquence est forte : même avec des produits qui ne sont pas 100 % « safe », tu peux réduire des frictions métier de façon très efficace sans exposer tes données — en utilisant l'IA pour fabriquer l'outil, pas pour mâcher la donnée. C'est la critique classique (« mais alors mes données partent ?! »), et elle est parfaitement mitigable.

    L'honnêteté qui va avec : ça tient tant que l'outil produit ne renvoie pas, lui, tes données vers un LLM distant (sinon on retombe dans le cas 1) ; et le code lui-même peut parfois être sensible. Mais bien posé, c'est l'une des façons les plus puissantes de concilier IA et confidentialité.

    7.5 Le risque, calibré (ni parano, ni naïf)

    Le vrai sujet, c'est quelle donnée tu exposes :

    • Anodin (brouillon, contenu public, template) → risque quasi nul.
    • Sensible (personnel, client, santé, juridique, secret professionnel) → sérieux : RGPD, nLPD (Suisse), secret pro (médecin, avocat, fiduciaire…), et hébergement souvent hors Suisse/UE.
    En une phrase

    Le risque = (quelle donnée) × (quelle formule + réglages) × (où c'est hébergé) — et souvent tu peux carrément éviter d'exposer la donnée en utilisant l'IA pour construire l'outil plutôt que pour traiter la donnée (§7.4).

    7.6 Hygiène de base (quel que soit ton niveau)

    • Jamais de secrets en clair (mots de passe, clés) dans un dossier ouvert à l'agent ; .gitignore-les.
    • Choisis le périmètre : un dossier dédié, pas tout ton disque.
    • Anonymise quand tu peux.
    • Désactive l'entraînement sur ton compte grand public ; pour du sérieux, offre business/API (pas d'entraînement) ou local.
    • Sépare comptes perso et pro.

    09 — La méthode

    Fais ton propre pack de contexte (la méthode)

    Tout ce qui précède devient concret au moment où tu donnes à l'agent ta matière : qui tu es, ton métier, tes contraintes, ta façon de faire. Ce bloc, c'est le pack de contexte — la mémoire durable que tu poses dans tes fichiers (§5) et que l'agent consulte. Voici la méthode généralisable pour le construire toi-même : exactement ce qu'un agent fait quand on le pilote, étape par étape.

    Si tu étais à l'atelier : tu repars avec ta version perso, déjà construite avec toi et taillée pour ton cas — ce guide-ci n'a pas à la refaire. La méthode ci-dessous est là pour tout lecteur (et pour toi, si tu veux la refaire à blanc, sur un autre sujet).

    9.1 Les six pas

    1. Choisis UN cas réel et non sensible — une tâche qui te bouffe du temps chaque semaine. Un seul, étroit, mesurable.
    2. Écris ton pack de contexte — dans un simple fichier texte : qui tu es, ton métier, tes contraintes, ton ton, tes règles, des exemples de ce que tu considères « bien fait ». C'est la mémoire que tu donnes à l'IA — celle qu'elle relira à chaque fois, au lieu que tu la recolles.
    3. Ouvre ton agent (Claude Code via ton abo, ou un autre, §6) sur un dossier dédié suivi par git — pour que chaque étape soit un point de sauvegarde réversible.
    4. Travaille en itératif : petits pas, tu valides à chaque tour, git garde l'historique → tu peux tout annuler sans rien casser.
    5. Structure ta doc dès que ça grandit (arborescence + index + journal de suivi, §4), pour que l'agent navigue sans tout relire.
    6. Stretch : connecte-le à une app que tu utilises déjà (API/MCP, §2) — et laisse-le souvent construire le pont lui-même.

    Quand tu touches du sérieux, relis les §7–§8 et monte d'un cran de protection avant d'ouvrir un périmètre sensible.

    9.2 Le réflexe

    Un pack de contexte n'est jamais « fini » : il s'enrichit à mesure que tu travailles. Chaque fois que tu reprends une réponse de l'agent (« non, chez moi on dit plutôt… »), note la règle dans le pack — la prochaine fois, l'agent l'aura déjà. C'est exactement ce mouvement, répété, qui transforme un chat anonyme en ton outil.

    En une phrase

    Commence petit, sur un cas sans risque, avec un pack de contexte écrit de ta main, en itératif — puis enrichis-le à chaque tour. C'est la méthode ; le reste, c'est de la répétition.

    10 — Le lexique

    Glossaire

    Les termes soulignés en pointillé dans le texte sont définis ici — dans la version web, clique dessus pour la définition.

    IA, modèles & agents
    • LLM (Large Language Model) — le moteur de texte derrière ChatGPT, Claude, etc. On lui a fait lire une quantité énorme de textes ; il a appris à prédire le mot suivant le plus probable, et c'est en enchaînant ces prédictions qu'il « écrit ». Il ne « comprend » pas comme un humain — il est juste extrêmement bon pour deviner la suite.
    • Modèle — le LLM lui-même, le « cerveau ». C'est souvent le même modèle qui répond dans un chat et qui anime un agent ; ce qui change, c'est ce qu'on met autour de lui (voir Agent).
    • Prompt — ta consigne, ce que tu écris à l'IA. La qualité de la réponse dépend énormément du prompt et du contexte que tu donnes — d'où l'intérêt d'un « pack de contexte » réutilisable plutôt que de tout réexpliquer à chaque fois.
    • Token — le « morceau de mot » que l'IA lit et écrit (≈ ¾ d'un mot ; ~750 mots ≈ 1000 tokens). C'est l'unité de tout : la longueur qu'un modèle peut traiter, et ce que tu paies à l'API. Plus c'est long, plus ça consomme.
    • Contexte / fenêtre de contexte — tout ce que l'IA « a sous les yeux » pour répondre : ton prompt + l'historique du fil + les fichiers fournis. C'est limité (une taille maximale) et temporaire (ça s'efface entre les sessions) — exactement pourquoi on externalise la mémoire dans des fichiers + git.
    • Paramètres (le « B », ex. 70B) — les réglages internes appris par le modèle, comptés en milliards (B). « 70B » = 70 milliards. En gros : plus il y en a, plus le modèle est capable… mais plus il est lourd et lent à faire tourner (surtout en local).
    • Inférence — le fait de faire tourner un modèle déjà entraîné pour obtenir une réponse. À ne pas confondre avec l'entraînement (la phase, très coûteuse, où il apprend). Quand tu « utilises » l'IA, tu fais de l'inférence.
    • Modèle « frontier » — les modèles les plus puissants du moment (Claude Opus, GPT-5.x, Gemini Pro…). Ils tournent dans le cloud sur d'énormes machines — impossibles à égaler sur un ordinateur personnel.
    • Open-weight (modèle ouvert) — un modèle dont le « cerveau » entraîné est public : tu peux le télécharger et le faire tourner toi-même, gratuitement, même hors ligne (Llama, Mistral, Qwen…). Moins puissant que le frontier, mais entièrement à toi.
    • Agent — le même modèle, mais mis au travail : on lui donne une boucle (planifier → agir → vérifier → recommencer), des outils (lire/écrire des fichiers, lancer des actions), un environnement (tes fichiers) et un objectif. Résultat : il fait le boulot, il ne se contente pas de répondre.
    • Boucle agentique — le cycle que répète un agent : il planifie une étape, agit, observe le résultat, corrige, et recommence — jusqu'à l'objectif, avec ta validation.
    Se brancher : API & connexions
    • API — la « façade de service » d'un logiciel : des portes par lesquelles un autre logiciel (ou un agent) peut lui demander des choses ou lui en envoyer. Presque toutes tes apps (CRM, compta, agenda, drive) en ont une.
    • Endpoint — une porte précise d'une API : un service donné, du type « donne-moi la liste des factures » ou « crée une fiche client ». Une API, c'est plein d'endpoints.
    • MCP (Model Context Protocol) — un standard (apparu fin 2024) qui sert de prise universelle entre un agent et tes outils : au lieu de bricoler une connexion sur mesure pour chaque app, on branche une fois, et l'agent sait s'en servir.
    • CLI / terminal — la fenêtre où l'on tape des commandes (pas de boutons, pas de souris). Ça fait « truc de dev », mais c'est juste une autre façon de piloter un programme — Claude Code, par exemple, y vit.
    • IDE / éditeur de code — le logiciel dans lequel les développeurs écrivent et organisent leur code (VS Code, Cursor…). Beaucoup d'outils agentiques s'y greffent.
    git & méthode de travail
    • git — un outil gratuit qui enregistre l'historique de tes fichiers, sur ta machine. Deux images : une machine à remonter le temps (chaque état est gardé) + un carnet de bord (qui a changé quoi, quand, pourquoi). C'est ce qui donne à l'agent une mémoire fiable.
    • Dépôt (repo) — un dossier suivi par git. L'unité de base : un projet = un repo.
    • Commit — un point de sauvegarde daté et commenté que tu poses dans l'historique. Tu peux toujours y revenir : c'est le « Ctrl+S » qui n'oublie jamais.
    • Versionning / historique — la suite des commits : chaque version est conservée, horodatée, attribuée, et tu peux revenir à n'importe laquelle. Rien ne se perd.
    • Branche — une ligne de travail parallèle : tu pars de l'état actuel, tu essaies un truc « à côté » sans toucher à l'original. Si ça marche tu fusionnes, sinon tu jettes — zéro risque.
    • Fusion (merge) — l'action de réunir une branche d'essai avec la version principale, une fois l'essai concluant.
    • .gitignore — un petit fichier où tu listes ce que git doit ignorer : mots de passe, clés, gros fichiers… Essentiel pour ne jamais envoyer un secret par erreur.
    • Réversibilité / traçabilité — deux superpouvoirs de git : revenir en arrière en un geste (réversibilité) et savoir exactement qui a changé quoi, quand, pourquoi (traçabilité). C'est ce qui supprime la peur de « tout casser ».
    • Itératif — avancer par petites boucles « fais un bout → vérifie → ajuste → continue », plutôt que tout d'un coup. C'est ainsi qu'un agent (et une équipe agile) travaille le mieux.
    • Agile — une manière de travailler par petits incréments livrables, avec du feedback fréquent, au lieu d'un grand projet figé d'avance. Les agents s'y prêtent naturellement.
    • Definition of done — se mettre d'accord, à l'avance, sur ce que veut dire « c'est fini » pour une tâche. Sans ça, un agent (comme un humain) ne sait pas quand s'arrêter.
    • Open source / propriétaire — open source = le code est public et réutilisable librement (ex. Aider, Cline) ; propriétaire (closed-source) = le code est fermé, contrôlé par l'éditeur.
    • BYO clé (BYOK, bring your own key) — l'outil est gratuit, mais tu y branches ta propre clé d'API : tu ne paies que la consommation du modèle, pas un abonnement.
    Documentation
    • Arborescence — l'organisation en dossiers et sous-dossiers. Bien rangée, elle permet à l'agent (et à toi) d'aller droit au bon fichier, sans tout parcourir.
    • Index — un sommaire qui oriente : « pour la facturation → tel fichier ». Il évite à l'agent de tout relire pour trouver une info (et économise temps et tokens).
    • Journal de suivi — un carnet de bord qui retrace, dans le temps, ce que l'agent a fait, quand, et ce qui a été livré. Il complète l'historique git : git dit ce qui a changé, le journal raconte les décisions.
    Coûts, modèles & matériel
    • GitHub / GitLab / Bitbucket… — des clouds (optionnels) où copier ton dépôt git pour le sauvegarder, le partager, collaborer. À ne pas confondre avec git, qui marche déjà tout seul en local.
    • Local vs cloud — local = ça tourne sur ta machine, rien ne sort. Cloud = ça tourne sur les serveurs d'un fournisseur, donc des données y transitent. Tout l'arbitrage confidentialité / puissance est là.
    • On-prem (on-premise) — « sur site » : chez toi, sur tes propres serveurs, pas dans le cloud d'un tiers. Contrôle maximal (et responsabilité maximale).
    • MTok (million de tokens) — l'unité de prix de l'API : un tarif par million de tokens, séparé pour ce que tu envoies (entrée) et ce que le modèle génère (sortie).
    • Pay-as-you-go — payer à la consommation réelle (l'API), sans abonnement. Idéal pour un usage irrégulier ou automatisé.
    • Batch (traitement par lot) — envoyer un paquet de requêtes en différé (réponse non immédiate) contre une grosse remise (≈ −50 %). Parfait pour du volume non urgent.
    • Mise en cache (prompt caching) — quand tu réutilises le même contexte d'une requête à l'autre, le fournisseur le garde « au chaud » et te le facture beaucoup moins (jusqu'à −90 %).
    • Quota — la limite d'usage incluse dans un abonnement avant blocage ou surcoût (ex. « 40 requêtes agent / mois »).
    • Crédits IA — une monnaie d'usage que certains outils consomment à la tâche (ex. GitHub Copilot : 1 crédit = 0,01 $). Tu paies ce que tu utilises, au-delà d'un quota inclus.
    • GPU — la carte graphique : le composant taillé pour faire tourner les modèles vite. GPU dédié = une vraie carte séparée (≠ la puce graphique intégrée, bien plus lente pour ça).
    • VRAM — la mémoire de la carte graphique. C'est elle qui décide quelle taille de modèle tu peux charger en local : pas assez de VRAM = le modèle ne rentre pas.
    • Mémoire unifiée — une mémoire partagée entre le processeur et le graphique (Mac Apple Silicon, AMD Ryzen AI Max+). Tu peux charger de très gros modèles sans carte dédiée — mais c'est moins rapide qu'un vrai GPU.
    • Bande passante mémoire — le débit (Go/s ou To/s) auquel la mémoire nourrit le processeur. En local, c'est elle qui fixe la vitesse de génération : beaucoup de mémoire mais peu de débit = gros modèle… mais lent.
    • HBM · GDDR · LPDDR · DRAM — des types de mémoire, du plus rapide au plus sobre : HBM (datacenter, ultra-rapide, ex. H100) > GDDR (cartes grand public) > LPDDR (portables, mémoire unifiée). DRAM = le terme général pour la mémoire vive.
    • Quantification (Q4, Q8…) — compresser un modèle pour qu'il tienne sur moins de mémoire et tourne sur du matériel modeste, au prix d'une légère perte de qualité. « Q4 » = compression à 4 bits (assez agressive, très utilisée en local).
    • MSRP — le prix catalogue conseillé par le fabricant. En période de pénurie (2026), le prix réel en magasin est souvent bien au-dessus.
    • Bedrock / Vertex / Azure — les plateformes cloud d'Amazon / Google / Microsoft pour faire tourner des modèles avec des conditions « entreprise » (pas d'entraînement sur tes données, région choisie, contrat).
    Données, confidentialité & droit
    • Entraînement — la phase où le fournisseur utilise des données pour améliorer ses modèles. L'enjeu pour toi : tes conversations servent-elles à ça ? (grand public : souvent oui, sauf opt-out ; API/entreprise : non).
    • Rétention — combien de temps le fournisseur conserve tes données après la requête (de quelques jours à plusieurs années selon l'offre et les réglages).
    • ZDR (Zero Data Retention) — une option (comptes entreprise éligibles) où rien n'est conservé après la réponse : la requête est traitée, puis oubliée.
    • Opt-out — l'action de refuser explicitement quelque chose — typiquement, refuser que tes conversations entraînent le modèle (un réglage à activer toi-même).
    • Offre business / entreprise — les formules professionnelles (≠ grand public). Différence clé : pas d'entraînement sur tes données par défaut, et des garanties contractuelles.
    • Anonymisation — remplacer les vraies données (noms, montants, identifiants) par des fictives avant de les envoyer à une IA. Le moyen le plus simple de profiter de l'outil sans exposer le sensible.
    • Huis clos — « en vase clos » : un outil (que tu as fait construire) qui traite tes données confidentielles sans que rien ne sorte vers un LLM distant. La clé pour réduire des frictions métier sans exposer la donnée (voir §7).
    • Contrat de sous-traitance (DPA) — le contrat qui encadre le traitement de données personnelles par un prestataire. Le RGPD / la nLPD l'exigent dès que des données perso sont en jeu.
    • Résidence des données — le pays / la région où tes données sont physiquement stockées (ex. UE vs États-Unis). Décisif pour la conformité (hébergement hors Suisse/UE = précautions).
    • Secret professionnel — l'obligation légale de confidentialité de certains métiers (médecin, avocat, fiduciaire, notaire…). Envoyer ces données à un LLM grand public peut le violer — d'où local / huis clos / cadre strict.
    • RGPD / nLPD — les lois de protection des données (RGPD côté UE, nLPD côté Suisse) : elles encadrent ce que tu as le droit de faire avec des données personnelles.
